# Принятие решений и управление конфликтами

---

## Принятие инженерных решений

### Decision-Making Framework

```
1. Сформулировать проблему
2. Собрать data / evidence
3. Сгенерировать варианты (≥3)
4. Оценить варианты по критериям
5. Принять явное решение
6. Задокументировать решение и rationale
7. Проверить результат через N времени
```

### Критерии оценки решений

| Критерий | Вопрос |
|----------|--------|
| **Impact** | Насколько сильно влияет на пользователей/бизнес |
| **Effort** | Сколько времени/ресурсов требует |
| **Risk** | Что может пойти не так |
| **Maintainability** | Насколько легко будет поддерживать |
| **Scalability** | Выдержит ли рост |
| **Reversibility** | Легко ли откатить |

### Reversible vs Irreversible Decisions (Jeff Bezos)

```
Type 1 (Irreversible)  — медленно, с review, с одобрением
Type 2 (Reversible)    — быстро, автономно, с правом на ошибку
```

> "Most decisions are Type 2. Use the fact that you can reverse them to move faster."

## Работа с неопределённостью

### Принципы

1. **Make the decision with the information you have** — ждать идеальных данных = паралич
2. **Set a deadline** — иначе решение никогда не будет принято
3. **Default to action** — если сложно выбрать, выберите то, что проще попробовать
4. **Prefer decisions you can undo** — reversible decisions → быстрее

### Decision Log (ADR — Architecture Decision Record)

```markdown
# ADR-001: Use PostgreSQL as primary database

## Status
Accepted

## Context
We need a primary data store for the order management system.
Requirements: ACID, complex queries, JSON support.

## Decision
Use PostgreSQL 16.

## Consequences
+ ACID compliance, rich query capabilities
- Need connection pooling (PgBouncer)
- Will need read replicas for scale

## Alternatives Considered
- SQL Server (license cost)
- Cosmos DB (eventual consistency not suitable for orders)
```

## Управление конфликтами

### Типы конфликтов

| Тип | Причина | Решение |
|-----|---------|---------|
| **Задача** | Разные мнения о решении | Data-driven обсуждение |
| **Процесс** | Как делать, а не что | Согласовать процесс заранее |
| **Межличностный** | Личные трения | Разделить людей и проблему |
| **Роли** | Кто за что отвечает | RACI-матрица |

### Модели разрешения

#### 1. Harvard Negotiation Model (Fisher & Ury)

```
     Position ("Я хочу NoSQL")           ← что говорят
        └── Interest ("Мне нужна гибкость схемы")  ← почему
             └── Need ("Быстрая итерация без миграций") ← глубинная причина
```

**Снимайте позицию, работайте с интересами.**

#### 2. DESC Model

```
D — Describe (конкретная ситуация)
E — Express (свои чувства/озабоченность)
S — Suggest (предложение)
C — Consequence (позитивный итог)
```

**Пример:**
> "Когда PR висит 3+ дня без комментариев (D), я беспокоюсь о скорости доставки (E). Предлагаю SLA на review — 4 часа для небольших PR (S). Это ускорит релизный цикл (C)."

#### 3. Интересы vs Позиции

```
Проблема: Выбрать базу данных
Позиция A: "PostgreSQL" 
Позиция B: "MongoDB"
Интересы A: ACID, SQL reporting
Интересы B: Flexible schema, fast writes

Решение: PostgreSQL + JSONB (ACID + гибкая схема)
```

### Когда эскалировать

| Признак | Действие |
|---------|----------|
| Не хватает полномочий | Эскалация с контекстом и вариантами |
| Нарушены ценности компании | Эскалация сразу |
| Deadlock (2 стороны не сдвигаются) | Решение от нейтрального тимлида |
| Ресурсы не выделяют | Data-driven кейс + impact бизнесу |

**Правило:** Сначала попробуй решить самостоятельно. Если не можешь — эскалируй **с контекстом и вариантами решений**, не просто «помогите».

## Переговоры (Negotiation)

### BATNA — Best Alternative To a Negotiated Agreement

```
Перед любыми переговорами определите:
- Что будет, если мы не договоримся?
- Каков мой запасной план?
- Каков запасной план другой стороны?
```

**Пример.** Обсуждаем дедлайн:
- Команда: "Нужно 4 недели"
- Менеджер: "Нужно через 2 недели"
- BATNA: "Можем сделать базовый functional prototype за 2 недели, полный функционал — за 4"

### Принципы переговоров для инженеров

1. **Будьте data-driven** — "Benchmark показал, что вариант A на 40% быстрее"
2. **Показывайте trade-offs** — "Если ускорим, получим technical debt"
3. **Pre-negotiate** — договоритесь о критериях до обсуждения вариантов
4. **Yes-and** — принимайте идею и развивайте, не отвергайте сразу

## Чек-лист

- [ ] Решения документированы (ADR)
- [ ] Для спорных решений — data, not opinions
- [ ] Конфликты решаются через интересы, не позиции
- [ ] Перед эскалацией — контекст + варианты
- [ ] BATNA известна до переговоров
- [ ] Type 2 решения принимаются быстро, без овербординга
- [ ] Retro после крупных решений — что сработало, что нет
