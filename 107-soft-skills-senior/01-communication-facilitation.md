# Коммуникация и фасилитация

---

## Почему это критично для Senior

> **"A senior engineer is not someone who knows more technical details, but someone who can make the team more effective."**

На уровне Senior ~50% работы — не написание кода, а **коммуникация**: обсуждения, код-ревью, документация, синхронизации, эскалации.

## Активное слушание

| Техника | Описание |
|---------|----------|
| **Paraphrasing** | «То есть вы предлагаете...», «Если я правильно понял...» |
| **Validation** | «Это хороший вопрос», «Я понимаю, почему это вызывает беспокойство» |
| **Clarifying** | «Не мог бы ты уточнить...», «Что именно ты имеешь в виду под...» |

**Факт:** В среднем люди говорят со скоростью ~150 слов/мин, но думают со скоростью ~400 слов/мин. Разрыв = отвлечение. Активное слушание заполняет этот разрыв.

## Обратная связь (Feedback)

### SBI-модель

```
S — Situation (конкретная ситуация)
B — Behavior (наблюдаемое поведение)
I — Impact (влияние на команду/проект)
```

**Пример плохо:**
> «Ты пишешь плохой код.»

**Пример хорошо (SBI):**
> «На вчерашнем код-ревью (S) ты трижды принял PR без единого комментария (B). Это привело к багу на стейджинге и задержке релиза (I). Давай обсудим, как улучшить процесс.»

### Radical Candor (Kim Scott)

```
         ┌──────────────┬──────────────┐
         │  Challenge   │              │
         │  directly    │  Radical     │
         │              │  Candor ◄─── │ ← цель
         ├──────────────┼──────────────┤
         │  Manipulative│  Ruinous     │
         │  Insincerity │  Empathy     │
         └──────────────┴──────────────┘
            Care personally ──────────►
```

## Фасилитация встреч

### До встречи

- Определить **цель** (decision, alignment, brainstorm?) и результат
- **Agenda** разослать за 24+ часа
- Определить **timebox** на каждый пункт
- Решить, кто **facilitator**, кто **decision-maker**

### Во время встречи

```
┌─────────────────────────────────────────────┐
│  1. Check-in (2 мин) — как пришли?          │
│  2. Контекст (5 мин) — что уже известно     │
│  3. Обсуждение (timebox) — по agenda        │
│  4. Decision / Next steps                   │
│  5. Check-out (2 мин) — резюме, кто что     │
│     делает и к какому сроку                │
└─────────────────────────────────────────────┘
```

### Техники фасилитации

| Техника | Когда | Как |
|---------|-------|-----|
| **Round-robin** | Доминирует один голос | Каждый говорит по очереди |
| **Dot voting** | Много вариантов | Каждый клеит точки на приоритеты |
| **Six Thinking Hats** | Спор / тупик | Разные перспективы (De Bono) |
| **Yes, and...** | Брейншторм | Принимаем и развиваем идеи |
| **Elevator pitch** | Слишком долго | 30 секунд на суть |

### После встречи

- Minutes — **решения** и **action items**, не стенограмма
- Action items: **Owner + Deadline**
- Ссылка на запись (если была)

## Аргументация и убеждение

### Структура аргумента

```
Claim   → "Нам нужно перейти на Saga-паттерн"
Reason  → "Потому что текущая distributed transaction блокирует строки на 5+ секунд"
Evidence→ "В последнем инциденте deadlock привёл к P0 на 40 минут"
Counter → "Да, это увеличит сложность кода, но Saga снизит coupling и улучшит scalability"
```

### Принципы убеждения для инженеров

1. **Data-driven** — метрики, цифры, факты
2. **Trade-offs** — показать, что понимаешь минусы
3. **Empathy** — понять боль стейкхолдера
4. **Incremental** — предложить шаги, а не всё сразу

## Письменная коммуникация

### RFC (Request for Comments) — документирование решений

```
Title: RFC-001: Migrate from Monolith to Microservices
Status: PROPOSED → IN REVIEW → APPROVED / REJECTED
Author: Ivan Petrov
Date: 2025-06-01

## Problem Statement
...

## Proposed Solution
...

## Alternatives Considered
...

## Trade-offs
...

## Decision
...
```

**Факт:** В крупных компаниях (Amazon, Uber) любой значимый change проходит через RFC/Design Doc с ревью от 2+ senior инженеров.
