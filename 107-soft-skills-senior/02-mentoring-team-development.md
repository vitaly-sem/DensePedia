# Менторство и развитие команды

---

## Почему менторинг — обязанность Senior

> "The true measure of a senior engineer is how many people they've helped grow to be better than themselves."

Менторинг — не опция, а **ожидаемая часть работы** Senior-инженера. Если вы не растёте команду, вы не Senior.

---

## Стили менторинга

### Situational Leadership (Hersey-Blanchard)

```
Уровень        │ Подход          │ Пример
junior         │ Directing       │ "Сделай X, вот как..."
middle         │ Coaching        │ "Что ты думаешь? А что если..."
senior         │ Supporting      │ "Я доверяю твоему решению, но давай проверим..."
lead           │ Delegating      │ "Ты отвечаешь за это, я на подхвате"
```

### Sponsored mentorship (спонсорство)

Не просто советуешь, а **открываешь возможности**:
- Рекомендуешь на интересные проекты
- Называешь имя в комнате, где принимаются решения
- Помогаешь получить видимость

### Инструменты менторинга

| Инструмент | Описание |
|------------|----------|
| **Pair programming** | Работа вместе над одной задачей |
| **Code review как обучение** | Не просто указать ошибку, а объяснить why |
| **Tech talks** | Внутренние доклады |
| **Shadowing** | Наблюдение за работой сеньора |
| **Stretch assignments** | Задачи чуть сложнее текущего уровня |

## Code Review как менторинг

### Плохой CR

> ❌ «Это неправильно. Переделай.»

### Хороший CR

```
❓ "Я вижу potential issue: при конкурентном доступе этот код может 
   вызвать race condition. Вот что я имею в виду — `lock` не защищает 
   чтение на строке 42. Предлагаю два варианта:
   1. Закрыть чтение тем же `lock`
   2. Использовать `ConcurrentDictionary`
  
   Что думаешь? Давай обсудим trade-offs."
```

### Правила конструктивного CR

1. **Сначала позитив** — отметьте, что сделано хорошо
2. **Вопросы, а не утверждения** — «Почему выбран этот подход?» vs «Это неправильно»
3. **Предлагайте альтернативы** — не просто укажите проблему
4. **Объясняйте why** — не ограничивайтесь what
5. **Разделите важное и косметическое** — блокирующие issues vs nit

## Карьерный growth команды

### Матрица компетенций

| Уровень | Код | Архитектура | Коммуникация | Менторинг |
|---------|-----|-------------|--------------|-----------|
| Junior | Пишет простые функции | Выполняет задачи | Отвечает на вопросы | Получает help |
| Middle | Пишет сложные фичи | Проектирует модули | Ведёт обсуждения | Помогает juniors |
| Senior | Задаёт стандарты | Проектирует системы | Фасилитирует встречи | Растит middle |
| Lead | Strategy | Enterprise | Влияет на org | Растит senior |

### 1:1 встречи (one-on-one)

```
Частота: раз в 1-2 недели
Длительность: 25-50 минут
Повестка (основная — за сотрудником):
├── Как дела? (energy, motivation)
├── Что мешает? (blockers)
├── Карьерный рост (skills, goals, feedback)
└── Что могу сделать я? (support needed)
```

**Факт:** Лучшие 1:1 — те, где **80% времени говорит сотрудник**, а не руководитель.

## Создание инженерной культуры

### Принципы высокой культуры

| Принцип | Практика |
|---------|----------|
| **Psychological safety** | Ошибки = обучение, не наказание |
| **Blameless post-mortem** | Системные причины, не люди |
| **Continuous improvement** | Ретроспективы, эксперименты |
| **Knowledge sharing** | RFC, tech talks, wiki |
| **High-quality bar** | Code review, testing, linting |

### Что убивает культуру

- **Hero culture** — когда проблемы решаются «геройством», а не системными улучшениями
- **Blame culture** — поиск виноватого вместо причины
- **Gatekeeping** — когда знания копятся в головах, а не документируются
- **Bus factor = 1** — критическая функциональность в руках одного человека

## Чек-лист Senior-ментора

- [ ] Регулярные 1:1 с каждым членом команды
- [ ] CR как обучение — объяснять why, не только what
- [ ] Документировать архитектурные решения (RFC/ADR)
- [ ] Делить знания: tech talks, wiki, pair programming
- [ ] Давать stretch assignments
- [ ] Следить за bus factor в команде
- [ ] Создавать psychological safety
